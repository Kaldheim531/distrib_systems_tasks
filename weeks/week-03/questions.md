# Ответы на вопросы по теме "API Gateway"

## 1. Чем отличается Forward Proxy от Reverse Proxy?

Forward Proxy стоит перед клиентом. Клиент отправляет запрос прокси-серверу, а прокси уже идет в интернет вместо клиента. Так можно скрыть IP клиента или настроить доступ через общий сервер.

Reverse Proxy стоит перед серверами. Клиент думает, что обращается к одному серверу, а на самом деле reverse proxy пересылает запрос нужному внутреннему сервису.

## 2. Что такое upstream в Nginx?

`upstream` — это описание бэкенда, куда Nginx будет отправлять запросы.

Например, можно назвать группу `notifications_backend` и указать внутри адрес сервиса: `127.0.0.1:8000`. Потом в `proxy_pass` можно писать не сам адрес, а имя upstream.

## 3. Зачем закрывать прямой доступ к микросервисам?

Так безопаснее. Клиент видит только API Gateway и не знает, какие сервисы стоят внутри.

Еще это удобнее для разработки: внутренние сервисы можно переносить, менять порты и количество копий, а внешний адрес для клиента остается тем же.

## 4. Какие задачи может выполнять API Gateway кроме маршрутизации?

API Gateway может проверять авторизацию, ограничивать количество запросов, собирать логи и метрики, кэшировать ответы, добавлять HTTPS и распределять нагрузку между несколькими копиями сервиса.

## 5. Что будет, если API Gateway упадет, и как от этого защититься?

Если API Gateway упадет, клиент потеряет доступ ко всем сервисам, которые были за ним спрятаны.

Чтобы защититься, обычно запускают несколько копий gateway, ставят перед ними балансировщик и настраивают автоматический перезапуск через Docker или Kubernetes.
