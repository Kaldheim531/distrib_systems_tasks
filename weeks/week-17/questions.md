# Ответы

## 1. Какие критерии использовались при разделении на сервисы?

Я разделил по задачам. Один сервис отвечает за товары, второй за уведомления. Так они не смешивают разную логику.

## 2. Почему выбран этот стек?

Python и FastAPI уже использовались в прошлых заданиях. Docker Compose выбран, потому что можно поднять все одной командой.

## 3. Как сервисы справляются с ошибками?

Если item не найден, сервис возвращает 404. Если сервис уведомлений упадет, товары все равно можно читать и создавать.

## 4. Как организован деплой без простоя?

В учебном варианте полного zero downtime нет. В Kubernetes это можно сделать через Deployment и rolling update.

## 5. Что было самым сложным?

Самое сложное — собрать несколько частей вместе: сервисы, gateway и docker compose.

## 6. Что бы улучшил за месяц?

Добавил бы базу данных, авторизацию, нормальные логи и gRPC между сервисами.
