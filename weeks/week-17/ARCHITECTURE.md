# Архитектура

Project code: items-s14

Проект состоит из двух микросервисов и gateway.

Первый сервис называется items-svc-s14. Он отвечает за items. В нем реализован полный CRUD: можно создать item, получить список items, получить один item по id, изменить item и удалить item. У item есть поля id, name и sku.

Второй сервис называется notify-svc-s14. Он принимает сообщения об уведомлениях и хранит их в памяти. Этот сервис сделан отдельно, чтобы логика товаров и логика уведомлений не смешивались.

Gateway сделан через nginx. Клиент обращается не напрямую к сервисам, а к gateway. Nginx смотрит на путь запроса и отправляет его в нужный сервис.

Если запрос идет на /api/items, nginx отправляет его в items-svc-s14.

Если запрос идет на /api/notify, nginx отправляет его в notify-svc-s14.

Снаружи используется REST API. Его удобно проверять через curl, браузер или Postman.

Для items добавлен gRPC. Контракт лежит в файле proto/items.proto. Из него сгенерированы файлы items_pb2.py и items_pb2_grpc.py. gRPC сервер запускается внутри items-svc-s14 на порту 50051.

Данные сейчас хранятся в обычных списках Python. После перезапуска контейнеров данные пропадают. Для учебного проекта этого достаточно, потому что главная цель здесь показать взаимодействие сервисов.

Обработчики FastAPI написаны через async def. Это простая асинхронность, чтобы сервер мог нормально обрабатывать запросы.

Логирование базовое. Uvicorn пишет HTTP-запросы, а в коде есть logging.info при создании item и отправке уведомления.

CI описан в .github/workflows/ci.yml. Там ставятся зависимости, запускаются тесты и собирается Docker-образ.

Порты проекта:

items-svc-s14 REST: 8161

items-svc-s14 gRPC: 50051

notify-svc-s14 REST: 8170

gateway: 8080

Запуск:

```bash
cd weeks/week-17
docker compose up --build
```

Проверка:

```bash
curl http://127.0.0.1:8080/health
curl http://127.0.0.1:8080/api/items
curl http://127.0.0.1:8080/api/notify
```

Остановка:

```bash
docker compose down
```
