# Ответы на вопросы по теме "Первый микросервис и REST API"

## 1. В чем разница между "ресурсом" и "коллекцией" в контексте REST API?

**Коллекция** — это множество однотипных ресурсов, например, "/tickets" — это коллекция всех билетов. Коллекция обычно возвращает список объектов.

**Ресурс** — это конкретный отдельный объект внутри коллекции, например, "/tickets/42" — это конкретный билет с идентификатором 42. Ресурс возвращает один объект.

Проще говоря: коллекция — это папка с файлами, а ресурс — это конкретный файл внутри папки.

## 2. Почему для создания объекта используется метод `POST` и код ответа `201`, а не `200`?

**POST** используется для создания нового ресурса на сервере, что соответствует смыслу метода (отправить данные на обработку/создание).

**Код 201 Created** более точен, чем 200 OK, потому что он:
- Указывает, что ресурс был успешно создан
- Содержит в заголовке Location URL нового ресурса
- Дает клиенту больше информации о результате операции

Код 200 OK слишком общий и не передает специфику создания нового объекта.

## 3. Что такое идемпотентность? Назовите HTTP-методы, которые являются идемпотентными.

**Идемпотентность** — это свойство операции, при котором многократное выполнение одного и того же запроса приводит к тому же результату, что и однократное выполнение. Другими словами, сколько раз ни отправляй запрос — эффект будет как от одного раза.

**Идемпотентные методы HTTP:**
- GET — сколько раз ни запрашивай, данные не меняются
- PUT — повторная отправка тех же данных не создаст новый ресурс, а обновит существующий до того же состояния
- DELETE — удаление уже удаленного ресурса ничего не меняет
- HEAD, OPTIONS

**Неидемпотентные методы:**
- POST — повторный запрос создаст новый ресурс (новый ID)

## 4. Почему ID объекта обычно генерируется на сервере и возвращается в ответе на `POST`?

**На сервере ID генерируется по нескольким причинам:**
1. **Уникальность** — сервер может гарантировать, что ID не повторятся
2. **Целостность данных** — клиенты могут ошибаться или намеренно указывать занятые ID
3. **Безопасность** — нельзя подобрать или предсказать ID
4. **Единый источник истины** — сервер отвечает за идентификацию ресурсов

**Возвращается в ответе**, чтобы клиент знал, какой ID присвоен созданному объекту, и мог потом обращаться к нему (например, для GET /tickets/{id} или DELETE).

## 5. Что такое "контракт API" и почему важно его соблюдать?

**Контракт API** — это формальное или неформальное соглашение между клиентом и сервером о том, как они взаимодействуют. Он определяет:
- Какие эндпоинты доступны
- Какие методы HTTP можно использовать
- Какой формат данных ожидается на входе (запрос)
- Какой формат данных будет на выходе (ответ)
- Какие коды ответа возвращаются в разных ситуациях

**Важно соблюдать контракт, потому что:**
1. Клиенты полагаются на него при разработке
2. Нарушение контракта сломает все приложения, которые используют API
3. Это позволяет разным командам разрабатывать клиент и сервер независимо
4. Обеспечивает предсказуемость поведения системы

## 6. Зачем команде разработчиков нужна спецификация, например, OpenAPI (Swagger)?

**OpenAPI (Swagger) нужна для:**
1. **Документации** — автоматически создается понятное описание API, которое всегда актуально
2. **Тестирования** — можно тестировать эндпоинты прямо из браузера (интерактивный Swagger UI)
3. **Генерации кода** — можно автоматически создать клиентские библиотеки на разных языках
4. **Согласования** — бэкенд и фронтенд разработчики могут обсуждать API по единой спецификации
5. **Валидации** — запросы и ответы автоматически проверяются на соответствие спецификации
6. **Онбординга** — новые разработчики быстро понимают, как работать с API

FastAPI автоматически генерирует OpenAPI спецификацию, что сильно упрощает разработку и документирование.
