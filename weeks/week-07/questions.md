# Ответы на вопросы по теме "gRPC"

## 1. Зачем нужен файл `.proto`? Почему мы не пишем классы сообщений сразу на Python?

Файл `.proto` нужен как общий контракт между клиентом и сервером.

Если описать сообщения только на Python, другие языки не смогут автоматически использовать этот контракт. А из `.proto` можно сгенерировать код для Python, Go, Java и других языков.

## 2. Что такое Unary RPC? Чем он похож на REST запрос?

Unary RPC — это один запрос и один ответ.

Клиент вызывает метод, например `GetReview`, отправляет одно сообщение и получает одно сообщение назад. По смыслу это похоже на REST-запрос, где клиент отправляет HTTP-запрос и получает HTTP-ответ.

## 3. Почему gRPC обычно эффективнее REST JSON по процессору и трафику?

gRPC использует Protocol Buffers. Это бинарный формат, он компактнее JSON и быстрее обрабатывается.

JSON — текстовый формат. Его удобно читать человеку, но машине нужно больше времени, чтобы разобрать строки, имена полей и типы.

## 4. Что такое обратная совместимость в Protobuf и зачем нужны номера полей?

Обратная совместимость означает, что новая версия контракта не ломает старых клиентов.

Номера полей, например `id = 1`, нужны для бинарного формата. Protobuf передает не имя поля, а его номер. Поэтому нельзя просто менять номера старых полей, иначе старый клиент начнет читать данные неправильно.

## 5. Что такое Stub на клиенте и Servicer на сервере?

Stub — это сгенерированный клиентский объект. Через него клиент вызывает методы сервера как обычные функции.

Servicer — это серверный класс. В нем разработчик реализует методы, которые описаны в `.proto`.

## 6. Когда выбрать REST, а когда gRPC?

REST удобен для публичных API, браузеров, простых сервисов и ситуаций, где важно легко смотреть запросы руками.

gRPC удобен для общения между внутренними микросервисами, где важны скорость, строгий контракт и генерация клиента из `.proto`.
