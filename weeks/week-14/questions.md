# Ответы на вопросы по теме "CI/CD"

## 1. В чем разница между Continuous Integration, Continuous Delivery и Continuous Deployment?

Continuous Integration — это автоматическая проверка кода: установка зависимостей, линтер, тесты.

Continuous Delivery — это когда после успешных проверок система собирает готовый артефакт, например Docker-образ, и его можно вручную выкатить.

Continuous Deployment — это когда после успешных проверок система сама выкатывает новую версию без ручного подтверждения.

## 2. Почему тесты должны запускаться до сборки Docker образа или хотя бы до его пуша?

Так мы не публикуем сломанный образ.

Если тесты упали, нет смысла тратить время на push в registry. Лучше остановить pipeline раньше.

## 3. Зачем нужно кэширование зависимостей в CI?

Без кэша CI каждый раз заново скачивает зависимости.

Кэш ускоряет pipeline. Например, `setup-python` может кэшировать pip-пакеты, и повторные запуски будут быстрее.

## 4. Что такое Secret в CI/CD?

Secret — это скрытая переменная для паролей, токенов и ключей.

Пароль от Docker Hub нельзя писать в YAML. Его кладут в GitHub Secrets, а в workflow используют как `${{ secrets.DOCKER_PASSWORD }}`.

## 5. Что такое Matrix Build?

Matrix Build — это запуск одного job в нескольких вариантах.

Например, можно проверить проект сразу на Python 3.10, 3.11 и 3.12. GitHub Actions сам создаст несколько запусков.

## 6. Почему нельзя хранить конфиги продуктовой среды в репозитории кода?

Потому что в прод-конфигах часто есть пароли, токены, адреса приватных сервисов и другие чувствительные данные.

Часть безопасных настроек можно хранить в репозитории, но секреты лучше хранить отдельно: в Secret Manager, GitHub Secrets или Kubernetes Secrets.
