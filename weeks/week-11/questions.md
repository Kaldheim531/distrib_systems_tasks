# Ответы на вопросы по теме "Docker Compose"

## 1. Зачем создавать пользовательскую сеть в Docker Compose, если есть дефолтная?

Пользовательская сеть делает инфраструктуру понятнее и изолирует сервисы проекта от чужих контейнеров.

Внутри такой сети контейнеры могут обращаться друг к другу по именам сервисов, например `redis` или `reviews-svc-s14`.

## 2. В чем разница между `build` и `image` в описании сервиса?

`build` говорит Compose собрать образ из Dockerfile.

`image` говорит использовать уже готовый образ. Например, `redis:7-alpine` или `nginx:1.27-alpine`.

Можно использовать оба поля: собрать образ через `build` и дать ему имя через `image`.

## 3. Что делает `depends_on`? Гарантирует ли он полную готовность контейнера?

`depends_on` задает порядок запуска контейнеров.

Обычный `depends_on` гарантирует только старт контейнера, но не гарантирует, что сервис внутри уже готов принимать запросы.

Чтобы ждать реальной готовности, используют `healthcheck` и условие `condition: service_healthy`.

## 4. Как работают healthcheck-и и как использовать их в `depends_on`?

`healthcheck` — это команда, которую Docker периодически запускает внутри контейнера.

Если команда завершается успешно, контейнер считается `healthy`. В `depends_on` можно указать `condition: service_healthy`, и тогда зависимый сервис будет ждать здорового состояния.

## 5. Как передать переменные окружения в контейнер через Compose?

Первый способ — через блок `environment`:

```yaml
environment:
  PROJECT_CODE: reviews-s14
```

Второй способ — через файл:

```yaml
env_file:
  - .env
```

## 6. Почему Docker Compose редко используют в продакшене?

Docker Compose удобен для локальной разработки и небольших стендов.

В продакшене чаще используют Kubernetes или похожие системы, потому что там нужны автоскейлинг, rolling update, восстановление после падений, секреты, балансировка и управление большим количеством серверов.
