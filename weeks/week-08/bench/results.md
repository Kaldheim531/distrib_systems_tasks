# Результаты замеров

Project code: sessions-s14

## Что сравнивалось

Сравнивались два варианта обращения к сервису `sessions`:

- REST: 1000 запросов к `GET /sessions`.
- gRPC: 1000 unary-вызовов `GetSession`.

## Результаты

| Вариант | Количество запросов | Время |
|---|---:|---:|
| REST | 1000 | 0.82 sec |
| gRPC | 1000 | 0.31 sec |

## Вывод

gRPC получился быстрее REST в этом простом замере. Основная причина в том, что gRPC использует бинарный формат Protocol Buffers и постоянный HTTP/2-канал.

REST проще запускать и проверять вручную через браузер или curl, но для внутреннего общения микросервисов gRPC обычно удобнее и быстрее.

## Streaming

В `.proto` добавлен server streaming метод:

```proto
rpc SubscribeSessions (SubscribeSessionsRequest) returns (stream SessionUpdate);
```

Клиент делает один запрос, а сервер может вернуть несколько сообщений подряд через поток.
