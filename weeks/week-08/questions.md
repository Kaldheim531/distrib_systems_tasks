# Ответы на вопросы по теме "gRPC Streaming и бенчмарки"

## 1. Что такое Server Streaming? Приведите пример использования.

Server Streaming — это когда клиент отправляет один запрос, а сервер возвращает не один ответ, а поток ответов.

Пример: клиент подписался на логи сервиса, а сервер отправляет новые строки логов по мере появления.

## 2. Чем Server Streaming отличается от получения большого списка в одном ответе?

Большой список приходит целиком одним ответом. Клиент ждет, пока сервер соберет все данные.

При streaming сервер может отправлять данные частями. Клиент начинает получать первые сообщения раньше и не обязан ждать весь список сразу.

## 3. Какие метрики производительности существуют?

Latency — задержка одного запроса. То есть сколько времени прошло от отправки запроса до ответа.

Throughput — пропускная способность. Обычно это количество запросов в секунду.

CPU Usage — нагрузка на процессор. Она показывает, сколько ресурсов тратит сервер или клиент.

Еще можно смотреть память, размер трафика и количество ошибок.

## 4. Почему важно переиспользовать Channel в gRPC клиенте при замерах?

Channel в gRPC — это соединение клиента с сервером.

Если создавать новый channel на каждый запрос, замер будет нечестным: мы будем измерять не только сам RPC, но и постоянное создание соединений.

Правильнее создать channel один раз и использовать его для многих вызовов.

## 5. В каких сценариях REST может оказаться быстрее или удобнее gRPC?

REST удобнее для публичных API, браузеров, простых интеграций и ручной проверки через curl.

REST может быть проще, если данные маленькие, запросов мало, а скорость не является главной проблемой.

gRPC лучше подходит для внутреннего общения микросервисов, где важны строгий контракт, скорость и streaming.

## 6. Что такое Head-of-Line Blocking и как HTTP/2 решает эту проблему?

Head-of-Line Blocking — это ситуация, когда один медленный запрос задерживает следующие запросы в очереди.

HTTP/2 помогает тем, что умеет передавать несколько потоков данных по одному соединению. Один медленный поток не должен полностью блокировать остальные.

gRPC работает поверх HTTP/2, поэтому может использовать это преимущество.
