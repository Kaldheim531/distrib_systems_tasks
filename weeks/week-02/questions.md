# Вопросы для самопроверки

1. Чем `PUT` принципиально отличается от `PATCH`? Какой из них является идемпотентным?
PUT полностью заменяет ресурс — клиент отправляет все поля, отсутствующие поля сбрасываются. PATCH обновляет частично — меняются только переданные поля. PUT идемпотентен (сколько раз ни отправляй запрос, результат одинаковый), PATCH может не быть идемпотентным (например, если увеличивает счетчик).
2. Какие HTTP коды ошибок вы знаете и в каких ситуациях их следует возвращать?
   - 400 vs 422
   - 401 vs 403
   - 404

400 Bad Request — проблема с синтаксисом запроса (кривой JSON, который нельзя распарсить). 422 Unprocessable Entity — данные распарсились, но не прошли валидацию (не тот тип, нет обязательного поля). 401 Unauthorized — пользователь не аутентифицирован (нет токена). 403 Forbidden — пользователь аутентифицирован, но у него нет прав. 404 Not Found — ресурс не существует.


3. Зачем нужны негативные тесты (тесты на ошибки)?

Негативные тесты проверяют, что система правильно реагирует на ошибочные ситуации. Пользователи ошибаются, хакеры пытаются сломать API, фронтенд может отправлять не те данные. Без негативных тестов узнаем о проблемах только в продакшене.


4. Почему важно возвращать ошибки в структурированном формате (например, JSON), а не просто текстом?

JSON-ошибки машиночитаемы — фронтенд может автоматически обработать их, подсветить конкретное поле. Все ошибки выглядят одинаково, клиент знает структуру. Можно передать дополнительную информацию: какое поле не прошло валидацию, какое значение ожидалось.

5. Что такое "обратная совместимость" API и как валидация помогает (или мешает) её поддерживать?
Обратная совместимость — новая версия API не ломает старых клиентов. Валидация помогает: четко определяет контракт, позволяет добавлять опциональные поля, отсеивает мусор. Мешает, если слишком строгая — нельзя добавить новые поля, изменение типа ломает клиентов. Нельзя менять типы, удалять поля, делать опциональные поля обязательными.
