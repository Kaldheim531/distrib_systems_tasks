# Security audit

Project code: items-s14

## Объект проверки

Проверялся учебный сервис `items`:

- REST resource: `/items`
- Service name: `items-svc-s14`
- Port: `8161`
- Extra field: `sku`

## Найденные проблемы

### 1. Нет аутентификации

Риск: High.

Сейчас любой клиент может отправить запросы на создание, изменение или удаление объекта.

Решение:

- добавить JWT или session-based authentication;
- проверять токен на каждом защищенном endpoint;
- возвращать `401`, если пользователь не вошел.

### 2. Нет авторизации на уровне объекта

Риск: High.

Если у объекта есть `id`, пользователь может попробовать обратиться к чужому объекту, просто поменяв число в URL.

Решение:

- хранить `owner_id` у каждого объекта;
- проверять, что текущий пользователь имеет право читать или менять объект;
- возвращать `403`, если прав нет.

### 3. Нет строгой валидации `sku`

Риск: Medium.

Поле `sku` должно иметь понятный формат. Сейчас можно отправить любую строку: пустую, слишком длинную или с мусорными символами.

Решение:

- задать минимальную и максимальную длину;
- разрешить только нужные символы;
- использовать Pydantic-модель и регулярное выражение.

Пример правила:

```text
sku: только A-Z, 0-9, дефис, длина 3-32 символа
```

### 4. Нет rate limiting

Риск: Medium.

Клиент может отправлять слишком много запросов и перегрузить сервис.

Решение:

- добавить rate limiting на gateway;
- ограничивать количество запросов по IP или user id;
- для Nginx использовать `limit_req`.

### 5. Нет HTTPS на внешнем доступе

Риск: Medium.

Если сервис доступен извне по HTTP, трафик можно перехватить.

Решение:

- включить TLS на gateway или ingress;
- хранить сертификаты вне кода;
- включить редирект HTTP -> HTTPS.

### 6. Контейнер может запускаться от root

Риск: Medium.

Если приложение внутри контейнера взломают, root-процесс даст атакующему больше возможностей.

Решение:

- создать отдельного пользователя в Dockerfile;
- запускать приложение через `USER app`;
- не давать контейнеру лишние capabilities.

## Примеры ручных проверок

SQL Injection:

```bash
curl "http://127.0.0.1:8161/items/1%20OR%201=1"
```

Длинный `sku`:

```bash
curl -X POST http://127.0.0.1:8161/items \
  -H "Content-Type: application/json" \
  -d '{"name":"test","sku":"AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA"}'
```

Командная инъекция:

```bash
curl -X POST http://127.0.0.1:8161/items \
  -H "Content-Type: application/json" \
  -d '{"name":"test","sku":"ABC; rm -rf /"}'
```

## Итог

Критичных проблем уровня SQL Injection или hardcoded secrets не найдено.

Главные риски для `items-s14` — отсутствие аутентификации, авторизации, rate limiting и строгой валидации входных данных.

Перед production нужно добавить защиту доступа, правила валидации, HTTPS, rate limiting и запуск контейнера от отдельного пользователя.
