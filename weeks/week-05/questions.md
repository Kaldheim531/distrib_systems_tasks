# Ответы на вопросы по GraphQL

## 1. Главное концептуальное отличие GraphQL от REST

В REST сервер решает, какие данные возвращать: каждый эндпоинт (`/users/1`, `/users/1/posts`) жёстко определяет форму ответа. Клиент получает то, что дают.

В GraphQL клиент сам описывает, что ему нужно, в теле запроса. Эндпоинт один — `/graphql` — но через него можно получить любую комбинацию данных за один запрос. Контроль над формой ответа переходит от сервера к клиенту.

---

## 2. Schema Definition Language (SDL)

SDL — это язык описания схемы GraphQL. Обычный текстовый файл (`.graphql`), в котором объявлены все типы данных, их поля и точки входа:

```graphql
type User {
  id: ID!
  name: String!
  email: String!
}

type Query {
  users: [User!]!
}

type Mutation {
  createUser(name: String!, email: String!): User!
}
```

SDL — это контракт между клиентом и сервером. Он не зависит от языка реализации: один и тот же `schema.graphql` можно реализовать на Python, Go или Node.js.

---

## 3. Query vs Mutation: отличия и параллелизм

**Query** — операция чтения. Не должна менять состояние сервера, как GET в REST.

**Mutation** — операция записи: создание, обновление, удаление данных, как POST/PUT/DELETE.

Ключевое отличие — **порядок выполнения**:
- Несколько `Query` в одном запросе выполняются **параллельно** — они не меняют данные, гонки нет.
- Несколько `Mutation` выполняются **последовательно** — чтобы одна не перезаписала результат другой.

---

## 4. Резолвер и кто его вызывает

Резолвер — это обычная функция, которая знает, *откуда* взять данные для конкретного поля. Например, резолвер для `users` идёт в базу и возвращает список пользователей, а резолвер для поля `email` просто читает его из объекта.

Вызывает резолверы **GraphQL-движок** (runtime). Когда приходит запрос, движок обходит дерево полей и для каждого вызывает соответствующий резолвер, передавая ему родительский объект, аргументы и контекст.

---

## 5. Проблема N+1 и DataLoader

**Проблема N+1** возникает, когда для списка из N объектов каждый связанный объект запрашивается отдельно. Например, получить 100 постов — 1 запрос, затем для каждого поста отдельно загрузить автора — ещё 100 запросов. Итого: 101.

**DataLoader** решает это через два механизма:
- **Batching** — вместо 100 отдельных `SELECT * FROM users WHERE id = ?` накапливает все нужные id и делает один `SELECT * FROM users WHERE id IN (1, 2, 3, ...)`.
- **Caching** — в рамках одного запроса уже загруженные объекты не запрашиваются повторно.

DataLoader работает как буфер: он собирает все вызовы за один "тик" event loop, а потом выполняет их одним батчем.

## 6. Почему HTTP-кэширование в GraphQL сложнее, чем в REST

В REST кэш завязан на URL и HTTP-метод: `GET /users/1` всегда возвращает одно и то же, и браузер или CDN легко это кэшируют по URL.

В GraphQL **все запросы идут через один эндпоинт** `POST /graphql`, а тело запроса у каждого разное. HTTP-кэш по умолчанию не смотрит в тело POST — он видит только URL, который всегда одинаковый. Значит, стандартные механизмы `Cache-Control`, CDN-кэш и браузерный кэш не работают из коробки.

Обходные решения:
- **Persisted Queries** — клиент отправляет хэш заранее известного запроса, сервер подставляет нужный текст. Запрос можно превратить в GET и кэшировать по хэшу.
- **Кэш на уровне приложения** — например, кэшировать результаты резолверов в Redis по ключу из аргументов.
- **Фрагментарный кэш** — кэшировать отдельные типы/объекты (например, через Apollo Client на фронтенде).
