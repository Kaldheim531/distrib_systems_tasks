# Ответы на вопросы по теме "GraphQL-клиент"

## 1. Зачем нужны `variables`? Почему нельзя склеивать строки?

`variables` нужны, чтобы передавать значения отдельно от текста запроса.

Если склеивать строку вручную через f-string, можно случайно сломать запрос или получить инъекцию. С переменными сам запрос остается одинаковым, а меняются только значения.

## 2. Из каких полей состоит стандартный JSON-запрос к GraphQL серверу?

Обычно запрос состоит из полей `query` и `variables`.

`query` — это текст GraphQL-запроса или мутации.

`variables` — это словарь со значениями для переменных из запроса.

Еще иногда добавляют `operationName`, если в одном тексте есть несколько операций и нужно выбрать одну.

## 3. Почему у GraphQL обычно только один URL endpoint?

В GraphQL обычно один endpoint, например `/graphql`, потому что нужное действие описывается внутри тела запроса.

В REST смысл часто находится в URL, например `/users/1`. В GraphQL URL один, а клиент сам пишет, какие данные хочет получить.

## 4. Что означает `errors` в ответе? Может ли оно прийти вместе с `data`?

`errors` означает, что при выполнении запроса произошла ошибка.

Да, `errors` может прийти вместе с `data`. Это значит, что часть запроса выполнилась, а часть нет.

Поэтому клиент должен проверять не только HTTP-статус, но и тело ответа.

## 5. Как обрабатывать ошибки, если HTTP статус 200?

В GraphQL статус `200 OK` может означать только то, что сервер принял HTTP-запрос.

После этого нужно прочитать JSON. Если там есть поле `errors`, надо обработать ошибки. Если есть `data`, можно использовать полученные данные.

## 6. В чем плюсы типизированных клиентов вроде Apollo или Relay?

Такие клиенты помогают работать с GraphQL в больших приложениях.

Они умеют кэшировать данные, проверять типы, обновлять интерфейс после изменений и удобнее обрабатывать запросы.

Для маленького скрипта достаточно обычного `httpx.post`, но для большого фронтенда типизированный клиент обычно удобнее.
