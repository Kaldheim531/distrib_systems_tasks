# Ответы на вопросы по GraphQL-клиенту

## 1. Зачем нужны `variables`? Почему нельзя склеивать строки?

Три причины:

**Безопасность.** Если подставлять значения через f-string, пользователь может передать строку, которая сломает запрос или изменит его логику — GraphQL-инъекция. С переменными значения никогда не попадают в текст запроса, сервер обрабатывает их отдельно.

**Производительность.** Сервер парсит и валидирует текст запроса при каждом вызове. Если текст всегда одинаковый (переменные снаружи), сервер может закэшировать результат парсинга и не делать это повторно.

**Читаемость.** Запрос с `$id: ID!` явно описывает свой интерфейс, его легче читать, тестировать и переиспользовать.

---

## 2. Из каких полей состоит стандартный JSON-запрос к GraphQL серверу?

```json
{
  "query": "query($id: ID!) { user(id: $id) { name email } }",
  "variables": { "id": "42" },
  "operationName": "GetUser"
}
```

- **`query`** — обязательное. Текст операции (query или mutation).
- **`variables`** — опциональное. Словарь с переменными, на которые ссылается запрос через `$`.
- **`operationName`** — опциональное. Нужно, если в тексте запроса несколько именованных операций — указывает, какую из них выполнить.

---

## 3. Почему у GraphQL обычно только один URL endpoint?

Потому что в REST URL несёт смысловую нагрузку: `/users/1/posts` — это ресурс. В GraphQL ресурсов как таких нет — есть граф данных. Вся маршрутизация происходит внутри тела запроса: клиент сам описывает, что хочет получить. Сервер не нужно учить новым путям при добавлении данных — достаточно расширить схему. Один эндпоинт `/graphql` — это и есть весь API.

## 4. Что означает `errors` в ответе? Может ли оно прийти вместе с `data`?

`errors` означает, что при выполнении запроса что-то пошло не так. Возможны три сценария:

| Ситуация | `data` | `errors` |
|---|---|---|
| Всё хорошо | ✅ есть | ❌ нет |
| Частичный успех | ✅ есть | ✅ есть |
| Полный провал | `null` или ❌ | ✅ есть |

Да, `data` и `errors` могут прийти одновременно — это частичный успех. Например, запрос просит пользователя и его посты: пользователь нашёлся, а посты упали с ошибкой. Клиент получит данные пользователя и ошибку для поля `posts`.

## 5. Как обрабатывать ошибки, если HTTP статус 200?

Нельзя полагаться только на HTTP статус — нужно всегда проверять тело ответа:

```python
result = response.json()

if "errors" in result:
    for error in result["errors"]:
        print("Ошибка:", error["message"])

if "data" in result:
    # обрабатываем данные, даже если были частичные ошибки
    process(result["data"])
```

Правило: `200 OK` в GraphQL означает лишь то, что HTTP-транспорт сработал. Бизнес-логика могла упасть — это покажет только `errors` внутри JSON.

## 6. Плюсы типизированных клиентов (Apollo, Relay) перед простыми HTTP-запросами

**Кэширование.** Apollo Client автоматически нормализует и кэширует ответы по типу и id объекта. Повторный запрос того же `User(id: 1)` не идёт на сервер — берётся из локального кэша.

**Типобезопасность.** Инструменты вроде GraphQL Code Generator читают схему и генерируют TypeScript-типы для каждого запроса. Опечатка в имени поля становится ошибкой компиляции, а не багом в рантайме.

**Реактивность.** Компоненты автоматически перерисовываются при обновлении данных в кэше. Не нужно вручную следить за состоянием.

**Оптимизации запросов.** Relay идёт ещё дальше: он статически анализирует все запросы в проекте и умеет объединять их, откладывать загрузку некритичных данных (`@defer`) и делать prefetch.

Простые HTTP-запросы (`httpx.post(...)`) проще и достаточны для скриптов или бэкенд-клиентов. Типизированные клиенты оправданы в больших фронтенд-приложениях, где важны производительность, консистентность данных и масштаб команды.ы
